## Profile

**Technical Background:** Same as STK-001 — experienced across Python, Docker, MQTT, MariaDB, Linux — but operating with degraded immediate context. The technical skills are intact; what is missing is current knowledge of the system's state, recent changes, and outstanding issues.

**Typical Interaction with System:** Episodic. Returns to the project after an absence of months, triggered by a new idea, a suspected problem, or a seasonal analysis goal. The first task is always orientation: establishing what version is deployed, what state the codebase is in, and what was left unfinished.

**Operating Constraints:**
- Same time constraints as STK-001: irregular availability, high opportunity cost for wasted orientation time.
- Operating from cold context — must reconstruct understanding quickly from available artefacts.
- Cannot rely on memory; must rely entirely on documentation, version history, and code clarity.

---

## Goals

- Know immediately and unambiguously which version is running in production.
- Re-orient to the current state of the codebase in minutes, not hours.
- Find the backlog of known improvements and add new ideas without losing either.
- Make a change confidently, knowing the test suite and documentation will catch regressions.

---

## Pain Points

- No clear record of what version is deployed or how recent the codebase is relative to production.
- No concise re-orientation guide; must read full documentation or reverse-engineer from code.
- New ideas from the current session displace memory of ideas from the last session because there is no persistent backlog.
- Unclear what is "done," what is "known but deferred," and what is "not yet considered."

---

## Resistance Factors

- A repository that is ambiguous about its own state: uncommitted changes, no tags, no CHANGELOG.
- Documentation that describes what the code does rather than how to get it running and where things stand.
- A codebase that has drifted from its specification — where the spec says one thing and the code does another — with no record of why.

---

## Current Workaround

Re-reading the codebase from scratch. Checking git log to reconstruct recent history. Running the service locally and inspecting output to infer current behaviour. All of this takes time that could be spent on the actual improvement that triggered the return.
