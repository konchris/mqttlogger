# Empathy Map: Future Maintainer

**Stakeholder ID:** STK-002
**Date:** 2026-05-08
**Note:** Inferred from STK-001 responses. Not directly elicited.

---

## Think & Feel

Returns to the project with a specific goal in mind — usually a new idea or a nagging suspicion that something is wrong — but immediately faces the overhead of re-orientation. Feels the tension between wanting to act on the new idea and needing to re-establish context first. Worries: "What did I leave unfinished last time? Is what's running in production the same as what's in the repository? What were the known issues?" Deeply values any artefact that collapses the time between returning and being productive.

## Hear

Echoes of past decisions: git commit messages, spec documents, a CHANGELOG if one exists. In the absence of these, must reconstruct intent from code alone — a slower and less reliable process. Has no external colleagues to ask; the only source of institutional knowledge is the documentation that past-self left behind.

## See

A repository in an uncertain state: maybe uncommitted changes, maybe a tag, maybe not. A `quickstart.md` that may or may not reflect the current setup. Log files from the last time the system was touched. A database with data in it, but no clear record of when the schema last changed or what sensors are currently expected to be present.

## Say & Do

Returns to the repository. Runs `git log`. Tries to remember how to get the dev environment running. Checks `docker compose ps`. Reads the last few commit messages to reconstruct what was in progress. If a quick-start document exists, reads it first. If a backlog exists, reads that to see what was deferred. Then — and only then — starts on the new idea.

## Jobs to Be Done

| Job Type | Job Statement |
|----------|---------------|
| Functional | Determine which version is deployed in production without ambiguity |
| Functional | Re-orient to the current system state within a single work session |
| Functional | Find the list of known improvements and add a new idea without losing either |
| Emotional | Feel that past-self left things in good order — that returning is a continuation, not an excavation |
| Social | Be consistent with the standards and intentions that past-self set out |

---

## Summary

STK-002's single most important need is **fast, reliable re-orientation** — knowing exactly where things stand without reverse-engineering it. A clean version history, a current quick-start document, and a lightweight backlog are not nice-to-haves; they are the difference between a productive return and a session spent reconstructing context that should have been recorded.
