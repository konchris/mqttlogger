# Empathy Map: Developer / Operator / Owner / Data Consumer

**Stakeholder ID:** STK-001
**Date:** 2026-05-08

---

## Think & Feel

Wants to trust the data. The deepest concern is not whether the service is running, but whether the record is complete and clean — because the entire value of the system rests on the integrity of the historical dataset. Knows from experience that silent failure is real and has already happened. Feels mild but persistent unease during periods of inattention: "Is it still logging? Has something changed upstream that I haven't noticed?" Aspires to the point where that unease is replaced by confidence grounded in observable evidence. Cares about building something that future-self will respect — not just something that works now.

## Hear

The broader home automation community emphasises "set and forget" reliability, but also increasingly values observability and data quality as systems mature. Hears from the Python/Docker ecosystem that good tooling (CI, linting, structured logs) is accessible and worth the setup cost. Has internalised from past experience that stub tests and invisible failures cost more to recover from than they save in the short term (the prior incident with `assert False` is the clearest example).

## See

A service running in a Docker Compose stack with no visible output when everything is working. Sensor readings appearing (or not appearing) in the database when queried directly through PyCharm Pro. Log files that confirm events happened, but no dashboard, no summary, no trend view. A HomeMatic installation that publishes good readings most of the time but occasionally floods with startup zeros. A growing database table whose completeness cannot be confirmed without manual inspection.

## Say & Do

Says: "It's set and forget." Does: manually inspects container status and runs ad-hoc SQL queries when a doubt surfaces. The gap between stated behaviour ("I don't really interact with it") and actual behaviour (periodic manual verification) reveals that trust in the system is not yet fully established. Invests time now in the SE baseline specifically because returning to undocumented systems has been painful in the past.

## Jobs to Be Done

| Job Type | Job Statement |
|----------|---------------|
| Functional | Capture every sensor reading reliably so that seasonal and renovation analyses are possible |
| Functional | Detect when the record is incomplete before an analysis depends on data that isn't there |
| Functional | Identify and exclude spurious HomeMatic startup zeros from analysis datasets |
| Emotional | Feel confident that the historical record is trustworthy — not just "probably fine" |
| Emotional | Feel that time spent on this project produces lasting value, not recurring re-work |
| Social | Be the kind of developer who builds systems that are clean, documented, and trustworthy |

---

## Summary

STK-001's central need is not functionality — the system already captures data. The need is **trust**: confidence that the record is complete, the failures are visible, and the system is comprehensible to its own builder after a long absence. Every improvement that increases observability, surfaces silent failures, or reduces re-orientation time directly serves this stakeholder's highest-priority goal.
